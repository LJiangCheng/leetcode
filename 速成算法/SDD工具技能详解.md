## 一、Spec-Kit（GitHub 官方）

> **定位**：GitHub 官方开源的 SDD 工具包，通过"规范→计划→任务→实现"的结构化流程，将 AI 从"代码生成工具"转变为"软件开发伙伴"。
>
> **依赖**：Python ≥ 3.11 + uv 包管理器
>
> **命令前缀**：`/speckit.*`

### 核心斜杠命令（5个）

#### 1. `/speckit.constitution` — 制定项目宪章

| 项目 | 详情 |
|------|------|
| **阶段** | 第一步（最高准则定义） |
| **是否必选** | 是 |
| **输出文件** | `.specify/memory/constitution.md` |

**详细描述**：

定义项目的最高准则，确保 AI 生成的所有内容符合团队规范。宪法是层级最高的文档，解决 AI 编程中"每次对话无状态"的问题——AI 每次调用都会先读取此文件，确保所有技术决策符合其中原则。

**定义内容包括**：
- **架构原则**：如"库优先原则"、"简洁性优先"
- **编码规范**：语言版本、格式化工具、测试框架
- **质量门控**：实现前必须通过的检查项
- **修订机制**：如何正式修改宪法内容

**示例**：
```
/speckit.constitution 建立专注于代码质量、测试标准和性能要求的项目原则。
要求采用TDD方式开发，代码覆盖率不低于80%。使用SQLite作为本地存储方案。
```

---

#### 2. `/speckit.specify` — 定义功能需求规格

| 项目 | 详情 |
|------|------|
| **阶段** | 第二步（需求规格化） |
| **是否必选** | 是 |
| **输出文件** | `specs/<编号>-<功能名>/spec.md` |

**详细描述**：

将模糊的用户需求转化为可测试、无歧义的功能规格。自动创建对应的 Git 特性分支（如 `001-photo-albums`）。

**输出内容包含三大核心部分**：

| 部分 | 说明 | 示例 |
|------|------|------|
| **用户故事** | 按优先级排列，每个故事必须可独立测试和交付 | "作为用户，我希望能创建任务，输入任务名称、优先级、截止日期" |
| **验收场景** | Given/When/Then 格式，描述可量化的成功标准 | "Given 用户已登录, When 点击创建按钮, Then 任务应保存并显示在列表中" |
| **边界条件** | 异常和边界情况处理 | "当截止日期为空时，默认不设置截止日期" |
| **非功能需求** | 性能、安全、可访问性等 | "API 响应时间 < 100ms" |

**模板内置 `[NEEDS CLARIFICATION]` 标记**，标示模糊点待后续澄清。与传统 PRD 的区别：`spec.md` 是写给 AI 执行的可执行规范。

---

#### 3. `/speckit.plan` — 生成技术方案

| 项目 | 详情 |
|------|------|
| **阶段** | 第三步（技术方案规划） |
| **是否必选** | 是 |
| **输出文件** | `plan.md`、`research.md`、`data-model.md`、`contracts/`、`quickstart.md` |

**详细描述**：

基于功能规范，AI 自动生成详细的技术实现计划。AI 会创建多个 SubAgent 并行查询互联网资料进行技术调研。

**输出文件清单**：

| 文件 | 内容 | 是否必选 |
|------|------|----------|
| **plan.md** | 技术上下文、宪法检查、项目结构、实现阶段、复杂度追踪 | 是 |
| **research.md** | 技术选型调研报告 | 否 |
| **data-model.md** | 数据表/模型的结构定义（字段、类型、约束等） | 否 |
| **contracts/** | 接口契约文件 | 否 |
| **quickstart.md** | 项目本地环境设置指南 | 否 |

**plan.md 核心内容**：
- **技术上下文**：编程语言版本、主要依赖、存储方案、测试框架
- **宪法检查（Constitution Check）**：验证计划是否符合架构原则
- **项目结构**：源码目录布局
- **实现阶段**：按用户故事组织的阶段划分
- **复杂度追踪**：记录架构决策理由

---

#### 4. `/speckit.tasks` — 分解任务列表

| 项目 | 详情 |
|------|------|
| **阶段** | 第四步（任务拆解） |
| **是否必选** | 是 |
| **输出文件** | `specs/<编号>-<功能名>/tasks.md` |

**详细描述**：

AI 将技术方案自动分解为可执行的任务单元。每个任务有唯一编号（如 T001、T002），按阶段组织。

**每个任务包含**：
- 前置条件
- 执行步骤
- 依赖关系
- **`[P]` 标记**：表示该任务可与其他任务并行执行，支持多 Agent 并行工作

**示例输出**：
```markdown
## 阶段1：项目初始化
- [ ] T001 按实现计划创建项目结构
- [ ] T002 [P] 配置代码检查与格式化工具

## 阶段2：基础设施
- [ ] T003 创建PostgreSQL消息表schema
- [ ] T004 [P] 配置Redis在线状态连接
```

---

#### 5. `/speckit.implement` — 执行任务生成代码

| 项目 | 详情 |
|------|------|
| **阶段** | 第五步（代码实现） |
| **是否必选** | 是 |
| **输出** | 项目源代码 |

**详细描述**：

AI 逐一执行任务列表中的任务，自动编写代码、创建文件、运行测试，并实时报告进度。

**关键特性**：
- 每完成一个任务，在 `tasks.md` 中打勾标记
- 支持 **"yolo 模式"**（全程托管）：AI 自动执行所有任务，开发者只需最终审查
- 每完成一个阶段后自动校验是否满足对应用户故事的验收标准
- 遵循 `constitution.md` 中的架构约束
- 参考 `contracts/` 中的接口契约
- 开发者只需审查代码是否符合规范，无需手动编写重复代码

### 辅助质量保障命令（3个）

#### 6. `/speckit.clarify` — 消除需求模糊点

| 项目 | 详情 |
|------|------|
| **是否必选** | 否（建议在 `/speckit.plan` 前执行） |
| **输出** | 更新 `spec.md` 的 `Clarifications` 节 |

**详细描述**：

AI 主动识别规范中的模糊点，通过选择题让用户澄清。针对规范中的 `[NEEDS CLARIFICATION]` 标记逐一提问，交互式引导用户澄清模糊点，将澄清结果写回规范文档。减少后续因需求模糊导致的返工。

**示例**：若规范中"任务过滤"描述模糊，AI 会提问：
> "任务过滤是否支持多条件组合（如'高优先级+截止日期本周'）？"

用户选择后，答案被写入规范。

---

#### 7. `/speckit.analyze` — 检查一致性与完整性

| 项目 | 详情 |
|------|------|
| **是否必选** | 否 |
| **输出** | 一致性分析报告 |

**详细描述**：

分析规范、计划、任务之间的一致性，生成一致性报告。检查维度包括：技术方案是否覆盖了所有功能需求、任务是否遗漏了依赖关系等。

**示例**：若计划中的"数据模型"未包含"任务创建时间"字段，而规范的"成功标准"要求"显示任务创建时间"，AI 会提示：
> "需补充 'created_at' 字段到数据模型"

---

#### 8. `/speckit.checklist` — 生成需求满足检查清单

| 项目 | 详情 |
|------|------|
| **是否必选** | 否（建议在 `/speckit.specify` 之后、`/speckit.plan` 之前运行） |
| **输出文件** | `.specify/specs/{分支名}/checklists/requirements.md` |

**详细描述**：

基于规范生成需求满足检查清单，辅助人工审查。被比喻为"用英语写的单元测试"——验证规范本身是否合格。

**检查项包括**：
- 所有 `[NEEDS CLARIFICATION]` 标记是否已消除
- 验收标准是否可量化、可测试（而非"用户体验良好"这类模糊描述）
- 是否覆盖了边界条件和异常流程
- 功能范围是否清晰界定，没有歧义
- 非功能需求（性能、安全等）是否明确

### CLI 命令

| 命令 | 说明 |
|------|------|
| `specify init <项目名> --ai <AI代理>` | 新建项目并初始化 Spec-Kit 结构 |
| `specify init . --ai <AI代理>` | 在当前目录初始化 |
| `specify init --here --force --ai <AI代理>` | 强制覆盖已有目录 |
| `specify init <项目名> --ai <AI代理> --no-git` | 跳过 git 初始化 |
| `specify check` | 验证安装是否成功 |
| `specify --version` | 查看版本信息 |

### Spec-Kit 工作流程

```
specify init                ← 初始化项目
    ↓
/speckit.constitution       ← 建立项目宪法
    ↓
/speckit.specify            ← 生成功能规范 → 审核 spec.md
    ↓
/speckit.clarify（可选）    ← 澄清模糊点
    ↓
/speckit.checklist（可选）  ← 检查规范质量
    ↓
/speckit.analyze（可选）    ← 分析文档一致性
    ↓
/speckit.plan               ← 生成技术方案 → 审核 plan.md 等
    ↓
/speckit.tasks              ← 分解任务列表 → 调整 tasks.md
    ↓
/speckit.implement          ← 执行代码实现
```

**五大核心设计思想**：

| 设计思想 | 体现 |
|----------|------|
| **意图驱动** | 开发者只描述"做什么"和"为什么"，不讨论"怎么做" |
| **模板约束** | 模板内置 `[NEEDS CLARIFICATION]`、验收标准格式、清单门控等约束 |
| **宪法治理** | `constitution.md` 定义架构原则，所有后续规划和实现必须遵循 |
| **分支即特性** | 每个特性在独立 Git 分支上开发，规范与代码共同版本化管理 |
| **并行可执行** | 任务列表中 `[P]` 标记支持多 Agent 并行工作 |

## 二、OpenSpec（Fission AI）

> **定位**：由 Fission AI 开发的轻量级 SDD 框架，用 `/opsx:*` 斜杠命令在仓库里生成"提案→规格→设计→任务"结构化文档，支持 25+ AI 编码助手。
>
> **依赖**：Node.js ≥ 20.19.0
>
> **命令前缀**：`/opsx:*`

### 核心模式命令（4个）

> 通过 `openspec config profile` 选择 `core` profile 激活

#### 1. `/opsx:explore [topic]` — 探索想法

| 项目 | 详情 |
|------|------|
| **参数** | `topic`（可选）— 要探索的主题 |
| **何时使用** | 需求不明确时 |
| **输出** | 无文件生成（纯讨论） |

**详细描述**：

让 AI 调查当前代码库，分析可行方案，提供选项比较，但不立即创建任何文档。适合在需求不明确时先进行讨论和调研。

**交互流程**：
1. AI 分析代码库
2. 提出多个方案选项
3. 用户选择方向
4. 建议下一步操作

**示例**：
```
/opsx:explore 移动端认证方案
/opsx:explore 如何优化数据库查询性能
```

---

#### 2. `/opsx:propose [change-name]` — 创建变更提案

| 项目 | 详情 |
|------|------|
| **参数** | `change-name`（可选）— 变更名称或描述 |
| **何时使用** | 开始新功能 |
| **输出** | 完整的变更规范文件夹 |

**详细描述**：

一键创建完整的变更规范文件夹，包含以下四个文档：

| 生成产物 | 说明 |
|----------|------|
| `proposal.md` | 为什么要做这个变更，变更内容是什么（动机、目标、范围、风险、回滚计划） |
| `specs/spec.md` | 需求和场景描述（Given/When/Then 格式） |
| `design.md` | 技术方案/架构设计（架构、组件接口、数据模型、API设计、安全考虑） |
| `tasks.md` | 实施任务清单（带文件路径和测试、完成标准） |

**示例**：
```
/opsx:propose add-dark-mode
AI:  Created openspec/changes/add-dark-mode/
     ✓ proposal.md
     ✓ specs/spec.md
     ✓ design.md
     ✓ tasks.md
     Ready for implementation!
```

---

#### 3. `/opsx:apply` — 实现任务

| 项目 | 详情 |
|------|------|
| **参数** | 无 |
| **何时使用** | 文档完成后 |
| **输出** | 项目源代码 |

**详细描述**：

自动读取当前变更的 `tasks.md` 任务列表，逐项实现代码。完成后报告所有任务状态。

**流程**：读取 tasks.md → 逐个实现 → 标记完成 → 汇报结果

**示例**：
```
/opsx:apply
AI:  Implementing tasks...
     ✓ 1.1 Add theme context provider
     ✓ 1.2 Create toggle component
     ✓ 2.1 Add CSS variables
     ✓ 2.2 Wire up localStorage
     All tasks complete!
```

---

#### 4. `/opsx:archive` — 归档变更

| 项目 | 详情 |
|------|------|
| **参数** | 无 |
| **何时使用** | 功能完成后 |
| **输出** | 归档文件夹 + 更新主规范 |

**详细描述**：

将完成的变更归档到 `openspec/changes/archive/` 目录（带日期前缀），同时更新主 spec 文档，保持工作区整洁。

**示例归档路径**：`openspec/changes/archive/2025-01-23-add-dark-mode/`

---

### 扩展模式命令（7个）

> 通过 `openspec config profile` 选择扩展配置，再运行 `openspec update` 激活

#### 5. `/opsx:new [change-name]` — 创建变更脚手架

**功能**：仅创建变更目录和 `.openspec.yaml` 元数据文件，不生成文档。适合想要逐步创建文档的用户。

---

#### 6. `/opsx:continue` — 逐个创建文档

**功能**：按照依赖关系，一次创建一个文档。显示当前文档的依赖状态和模板路径，生成后提示下一步。

**文档依赖链**：
```
proposal → specs → design → tasks → implement
```

---

#### 7. `/opsx:ff` — 快速创建所有规划文档

**功能**：一次性创建所有规划文档（proposal、specs、design、tasks），等同于 `/opsx:propose` 但在扩展模式下使用。跳过逐步创建流程。

---

#### 8. `/opsx:verify` — 验证实现

**功能**：验证代码实现是否符合文档规范，检查维度包括：
- tasks.md 中的任务完成情况
- 代码实现正确性
- 测试覆盖情况

---

#### 9. `/opsx:sync` — 同步增量规范

**功能**：将增量变更的规范同步合并到主规范中，保持规范与代码的一致性，适合长期维护的项目。

---

#### 10. `/opsx:bulk-archive` — 批量归档

**功能**：一次性归档多个已完成的变更，支持按日期筛选。适合清理多个完成项。

**示例**：
```bash
openspec bulk-archive --before 2026-01-01
```

---

#### 11. `/opsx:onboard` — 引导教程

**功能**：提供端到端的工作流教程，帮助新手学习 OpenSpec 的完整使用方法。为新加入的 AI 或开发者提供项目上下文和规范概览。

---

### OpenSpec CLI 命令

| 命令 | 功能 |
|------|------|
| `openspec init` | 在项目目录中初始化 OpenSpec |
| `openspec update` | 刷新 AI 代理指令，确保最新斜杠命令生效 |
| `openspec config profile` | 选择工作流配置文件（core / custom） |

---

### OpenSpec 工作流程

**核心模式流程**：
```
/opsx:explore → /opsx:propose → /opsx:apply → /opsx:archive
  （可选）      （创建文档）    （实现代码）    （归档完成）
```

**扩展模式流程**：
```
/opsx:new → /opsx:continue → /opsx:apply → /opsx:verify → /opsx:sync → /opsx:archive
  创建       逐步创建文档      实施变更      验证结果       同步规范       归档完成
```

**文档模板结构**（每个变更包含 4 类文档）：

| 文档 | 文件 | 内容 |
|------|------|------|
| **变更提案** | `proposal.md` | 动机、目标、范围、风险、回滚计划 |
| **规格说明** | `specs/spec.md` | 用户场景(Given/When/Then)、功能需求、非功能需求 |
| **技术设计** | `design.md` | 架构、组件接口、数据模型、API设计、安全考虑 |
| **实现任务** | `tasks.md` | 任务清单(带文件路径和测试)、完成标准 |

**核心哲学**：

| 原则 | 说明 |
|------|------|
| 灵活而非僵化 | fluid not rigid — 无刚性阶段门控，随时更新任何文档 |
| 迭代而非瀑布 | iterative not waterfall — 支持渐进式开发 |
| 简单而非复杂 | easy not complex — 轻量级 spec 层 |
| 为现有项目设计 | built for brownfield — 不仅适用于全新项目 |
| 从个人到企业级 | scalable — 可扩展 |

## 三、Superpowers（Agentic Skills 框架）

> **定位**：基于技能（Skills）的 AI 编程代理框架，通过一套可组合的技能和标准化工作流程，让 Claude Code、Codex 等编码助手能够像专业开发团队一样工作。核心理念：**Process over Prompt**——用流程规范代替 prompt 调优。
>
> **依赖**：AI 编程工具本身（Claude Code / Cursor / Codex CLI / Gemini CLI 等）
>
> **不是独立 CLI**，而是寄生在 AI 编程工具里的插件

### 协作类 Skills（9个）

#### 1. brainstorming — 头脑风暴

| 项目 | 详情 |
|------|------|
| **分类** | 协作类 |
| **作用** | 需求澄清与方案设计的第一步，强制在写代码前把需求想清楚 |
| **工作流位置** | 7步流程第1步 |

**具体行为**：
- 探索项目上下文
- **逐个提问**澄清需求（刻意设计为逐个提问，而非一次性甩出所有问题，避免用户随便应付）
- 提出 2-3 个方案
- 分段展示设计文档供用户审阅

**硬性门槛**：**不展示设计并获得用户批准前，绝不启动任何实现**

**老项目特殊指导**（Working in existing codebases）：
- 遵循现有代码模式和架构
- 不提议无关的重构
- 新代码要和项目风格一致（"入乡随俗"）

---

#### 2. writing-plans — 编写计划

| 项目 | 详情 |
|------|------|
| **分类** | 协作类 |
| **作用** | 将设计文档拆分为可执行的小任务 |
| **工作流位置** | 7步流程第3步 |

**具体行为**：
- 把设计文档拆成 **2-5 分钟**的小任务
- 每个任务包含三样东西：**精确的文件路径**、**完整的代码**（禁止 TBD、TODO 这类占位符）、**验证步骤**

**任务粒度原则**：太大子代理容易跑偏，太小上下文切换成本高，2-5分钟是官方推荐的经验值。

**产出**：计划文件，是后续子代理执行的依据。

---

#### 3. executing-plans — 执行计划

| 项目 | 详情 |
|------|------|
| **分类** | 协作类 |
| **作用** | 按照 writing-plans 产出的计划文件，逐步执行每个任务 |

**与 subagent-driven-development 的关系**：executing-plans 是计划执行的总控，subagent-driven-development 是具体的子代理派发机制。

---

#### 4. subagent-driven-development — 子代理驱动开发

| 项目 | 详情 |
|------|------|
| **分类** | 协作类 |
| **作用** | 为每个任务派发一个独立的子代理执行开发，实现上下文隔离 |
| **工作流位置** | 7步流程第4步 |

**核心思想**：每个子代理只看到当前任务所需的信息，**不会被之前任务的上下文污染**。

**具体行为**：
1. 为每个任务派发独立子代理
2. 子代理完成后要过**两道审查**：
   - **规范合规审查**：有没有违反设计文档
   - **代码质量审查**：代码风格、性能、安全
3. 审查不通过则打回去重做

---

#### 5. dispatching-parallel-agents — 并行代理调度

| 项目 | 详情 |
|------|------|
| **分类** | 协作类 |
| **作用** | 当有多个不相关的独立任务时，并行派发多个子代理同时处理 |

**适用场景**：多个不相关的独立任务可并行执行时使用。

**价值**：提高复杂项目中多任务场景的开发效率。

#### 6. requesting-code-review — 请求代码审查

| 项目 | 详情 |
|------|------|
| **分类** | 协作类 |
| **作用** | 自动发起代码审查流程 |
| **工作流位置** | 7步流程第6步 |

**具体行为**：
- 自动获取 git SHA
- 派发审查子代理
- 根据反馈修复问题

**审查维度**：代码质量、规范合规、潜在风险

**老项目特殊侧重**：新代码是否和项目风格一致 + 有没有引入不必要的依赖

#### 7. receiving-code-review — 接收代码审查

| 项目 | 详情 |
|------|------|
| **分类** | 协作类 |
| **作用** | 处理代码审查的反馈结果 |

**与 requesting-code-review 配合**：一个发起审查，一个接收处理审查结果。

#### 8. using-git-worktrees — 使用 Git Worktree 隔离

| 项目 | 详情 |
|------|------|
| **分类** | 协作类 |
| **作用** | 创建独立工作空间，与主分支隔离 |
| **工作流位置** | 7步流程第2步 |

**具体流程**：选择目录 → 验证 .gitignore → 创建 worktree → 运行项目配置 → 验证测试基线

**核心价值**：
- **代码隔离**：改坏了直接删掉重来，主分支干干净净
- **安全沙箱**：提供一个可以随时丢弃的沙箱，在老项目中尤为重要

**注意**：即使在老项目加小功能，也**不要跳过**这一步。

#### 9. finishing-a-development-branch — 完成开发分支

| 项目 | 详情 |
|------|------|
| **分类** | 协作类 |
| **作用** | 开发完成后的收尾工作 |
| **工作流位置** | 7步流程第7步 |

**前提条件**：验证所有测试通过

4. 丢弃分支

### 测试类 Skills（1个）

#### 10. test-driven-development — 测试驱动开发

| 项目 | 详情 |
|------|------|
| **分类** | 测试类 |
| **作用** | 强制执行 TDD 流程，确保代码质量 |
| **工作流位置** | 7步流程第5步 / 修 bug 3步流程第2步 |

**核心循环**：**RED → GREEN → REFACTOR**
- **RED**：先写一个会失败的测试
- **GREEN**：写能跑通的代码，越简单越好
- **REFACTOR**：清理代码，保持测试通过

**铁律**：**没有失败测试就不写生产代码**

**对 AI 编程的特殊价值**：
- 迫使每一步都只做够用的事，防止 AI 过度设计
- 迫使在写代码前思考接口设计
- 提供即时反馈，写完马上就能验证
- 形成安全网，后续改动不怕引入回归

**修 bug 时的要求**：先写一个能复现 bug 的失败测试，确认真的找到了根因。

### 调试类 Skills（2个）

#### 11. systematic-debugging — 系统化调试

| 项目 | 详情 |
|------|------|
| **分类** | 调试类 |
| **作用** | 系统化地发现和修复 bug，而非碰运气式修复 |
| **工作流位置** | 修 bug 3步流程第1步 |

**严格分四个阶段**：

| 阶段 | 名称 | 行为 | 目标 |
|------|------|------|------|
| 阶段一 | **根因调查** | 不做任何修改，只做信息收集（检查日志、复现bug、查看相关代码） | 找到bug的真正原因，而不是表象 |
| 阶段二 | **模式分析** | 分析bug模式（偶发/必现？特定输入/环境相关？特定用户/所有用户？） | 直接影响修复策略的选择 |
| 阶段三 | **假设与测试** | 基于前两阶段发现形成修复假设，写能复现bug的自动化测试验证 | 用测试验证假设是否正确 |
| 阶段四 | **实施修复** | 确认假设正确后才动手改代码，改完跑完整测试套件 | 确保修复有效且无回归 |

**铁律**：**不做根因调查就不修 bug**

---

#### 12. verification-before-completion — 完成前验证

| 项目 | 详情 |
|------|------|
| **分类** | 调试类 |
| **作用** | 在声明完成之前，必须提供新鲜的验证证据 |
| **工作流位置** | 修 bug 3步流程第3步 |

**"新鲜"的定义**：不是"昨天跑过了"，不是"之前没问题"，而是**刚刚验证过**。

**必须满足三个条件**：
1. 所有测试通过（包括新写的和原有的）
2. 手动验证 bug 已经修复
3. 相关功能没有回归

**铁律**：**没有新鲜验证证据就不做完成声明**

**对抗的问题**：改了A功能，B功能悄悄坏了但没测B
